## スコープ

- 対象: 採用済みの改善案

## 完了定義（Definition of Done）

1. 仕様が md に記載されレビュー済み
2. 実装が main へマージ済み
3. 回帰テスト（手動または自動）を通過
4. README/運用文書の更新完了
5. 既知制約とロールバック手順を記録

## 121/122 の実装指針

## 121 返信表示制限

- 方針:
1. 初期表示件数を定数化（例: 3件）
2. 残り返信は隠し、もっと見るで段階表示
3. 画面幅に応じて既定件数を調整可能にする

- 追加テスト:
1. 返信 0/1/3/10 件で表示崩れがない
2. もっと見る連打時の重複描画がない

## 122 通知の対象投稿可視化

- 方針:
1. 通知レコードだけでなく対象投稿本文（先頭）を表示
2. 取得不能時はフォールバック文言を表示
3. クリックで該当投稿へ遷移可能にする

- 追加テスト:
1. like/repost/reply/quote で期待どおり表示
2. 取得失敗時に UI が破綻しない

## リスクと対策

1. API 制約変更
- 対策: API 層を集約し呼び出し側を疎結合化

2. UI 複雑化による回帰
- 対策: 共通コンポーネント化と E2E 導入

3. 工数超過
- 対策: 各マイルストーンで Must/Should/Could を分離

2. 進捗は週次で「完了/進行中/未着手」を更新
3. ブロッカーは 24 時間以内に共有する
4. 仕様変更は本ファイルへ追記して履歴を残す
